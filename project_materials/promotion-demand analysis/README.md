# 促销需求分析：支持库存与渠道规划

## 1. 项目概述
本项目围绕促销期间的销售表现展开分析，目标是为库存配置和渠道规划提供数据支持。  
项目基于 PostgreSQL 数据库中的销售、客户、产品和经销商相关数据，识别区域需求差异、渠道表现、产品层级表现以及停产车型生命周期趋势。

最终交付将包括清洗后的分析数据集、SQL 提取与转换逻辑、结果校验过程，以及面向业务决策的 Tableau 可视化仪表盘。

## 2. 业务问题
本项目主要希望回答以下业务问题：

- 哪些区域在促销期间表现出更强的需求，值得优先配置库存？
- 不同销售渠道在不同价格层级下的表现有何差异？
- 哪些产品或产品层级更值得重点投入销售资源？
- 销售额和交易数量随时间如何变化？
- 已停产车型在经销商渠道中的累计销售表现如何？

## 3. 数据来源
本项目使用 PostgreSQL 数据库中的关系型业务表，主要包括：

- `sales`：交易级销售记录
- `customers`：客户地理信息
- `products`：产品及车型属性信息
- `dealerships`：经销商覆盖信息

这些原始表将被整合并转换为适合分析和可视化的中间数据集。

## 4. 数据提取
项目的第一步是识别与业务问题相关的核心表，并提取后续分析所需字段。

主要提取工作包括：
- 查看表结构与字段含义
- 确定表之间的关联键，如 `customer_id` 和 `product_id`
- 提取与交易、区域、产品、渠道有关的字段
- 根据分析目标限制数据范围

## 5. 数据理解
在正式清洗和建模之前，需要先对原始数据进行理解和初步检查。

主要检查内容包括：
- 日期范围覆盖情况
- 渠道字段的取值分布
- 地理字段和产品字段是否存在缺失值
- 销售金额的大致范围与异常情况
- 哪些字段适合用于过滤、分组和特征构造

## 6. 数据清洗与转换
为了让原始关系型数据能够直接用于业务分析，本项目将使用 SQL 和 Python 对数据进行清洗和转换，生成可直接用于 Tableau 的分析表。

主要清洗与转换步骤包括：
- 关联销售、客户、产品和经销商表
- 按分析需求过滤时间范围和交易范围
- 剔除不满足分析阈值的记录
- 构造衍生字段，如 `price_category`、`transaction_year`
- 为生命周期分析计算累计销售字段
- 对类别字段进行标准化处理，便于可视化展示

## 7. 分析数据集
项目计划生成多张清洗后的分析数据表，用于后续分析和 Tableau 仪表盘开发。

### A. `city_sales_analysis`
用于分析不同年份、渠道、价格层级和产品的销售表现。

预期字段包括：
- `product_id`
- `channel`
- `sales_amount`
- `price_category`
- `transaction_year`
- `sales_transaction_date`

### B. `cumulative_sales_dealership`
用于分析已停产车型在经销商渠道中的累计销售趋势。

预期字段包括：
- `model`
- `model_year`
- `product_type`
- `sales_transaction_date`
- `total_daily_sales`
- `cumulative_sales`
- `model_and_year`

### C. `state_model_sales`
用于分析促销期间各州与各车型的销售表现。

预期字段包括：
- `state`
- `model`
- `total_sales`

## 8. 数据校验
为了保证分析结果可靠，在进入最终可视化阶段前，需要对转换后的分析表进行校验。

计划进行的校验包括：
- 过滤前后行数检查
- 按产品或年份抽样核对销售总额
- 检查价格层级字段构造是否符合预期
- 确认 Tableau 使用的数据与 SQL 聚合结果一致

## 9. Tableau 仪表盘规划
最终的 Tableau 输出计划包含多个页面，用于支持不同层级的业务分析。

### 仪表盘 1：整体销售概览
主要展示：
- 总销售额
- 总交易数
- 平均销售额
- 分年份销售趋势
- 渠道销售对比
- 价格层级分布

### 仪表盘 2：区域与车型表现
主要展示：
- 各州销量表现
- Top / Bottom 州对比
- 车型筛选
- 促销期间区域表现比较

### 仪表盘 3：产品与生命周期趋势
主要展示：
- 累计销售趋势
- 不同车型年份对比
- 产品层级与趋势分析
